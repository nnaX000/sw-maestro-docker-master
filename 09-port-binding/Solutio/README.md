# 해설 — 문제 09

## 1. 세 가지 오해 정리

**오해 ① "EXPOSE 하면 포트가 열린다" → ✗**
`EXPOSE`는 "이 컨테이너는 이 포트를 쓴다"는 **문서/메타데이터**일 뿐입니다. 실제로 호스트에 포트를 여는 건 `docker run -p` 입니다. EXPOSE만 있고 `-p`가 없으면 호스트에서 접속되지 않습니다.

**오해 ② "-p 의 좌우 순서는 아무래도 된다" → ✗**
`-p 8080:5000` = **호스트 8080 → 컨테이너 5000**. 왼쪽이 호스트, 오른쪽이 컨테이너입니다. 컨테이너 앱이 5000에서 도는데 오른쪽을 80으로 적으면 연결되지 않습니다. 내부적으로는 iptables의 DNAT 규칙으로 호스트 포트로 온 패킷을 컨테이너로 돌려보냅니다.

**오해 ③ "-p 8080:5000 은 안전하다" → ✗**
`-p 8080:5000` 은 사실 `-p 0.0.0.0:8080:5000` 입니다. **모든 인터페이스(외부 포함)에 노출**됩니다. 클라우드 인스턴스라면 인터넷에서 접근 가능해질 수 있습니다. 로컬에서만 접근하게 하려면 `-p 127.0.0.1:8080:5000` 처럼 바인딩 주소를 명시하세요.

## 2. 문법 요약

| 문법 | 의미 |
|------|------|
| `-p 8080:5000` | `0.0.0.0:8080` → 컨테이너 5000 (외부 전부 노출) |
| `-p 127.0.0.1:8080:5000` | 로컬에서만 접근 가능 |
| `-p 5000` | 호스트 랜덤 포트 → 컨테이너 5000 |
| `docker port <c>` | 실제 매핑 확인 |

## 3. 실무 기준

- 외부에 보여줄 것(프론트/게이트웨이)만 `-p`로 노출하고, 내부 서비스(8번의 api, DB 등)는 **노출하지 않습니다**.
- 디버깅·관리 포트는 `127.0.0.1:` 로 묶어 외부에서 못 닿게 합니다.
- "열 필요가 없는 포트는 열지 않는다"가 원칙입니다.
