# 해설 — 문제 01

## 1. Dockerfile 정답

`app/Dockerfile`로 아래 내용을 작성합니다.

```dockerfile
# 1) 베이스 이미지 — Go 컴파일러가 들어있는 풀 이미지
FROM golang:1.22

# 2) 작업 디렉터리 지정 (이후 명령의 기준 경로)
WORKDIR /app

# 3) 모듈 정의 먼저 복사  (소스보다 먼저! 캐시 때문)
COPY go.mod .
# 의존성이 있다면 여기서 받음 (이번엔 표준 라이브러리뿐이라 사실상 없음)
RUN go mod download

# 4) 그다음 소스 복사 → 빌드
COPY . .
RUN go build -o server .

# 5) 컨테이너가 쓰는 포트 명시 (문서/메타데이터)
EXPOSE 5000

# 6) 컨테이너 시작 시 자동 실행할 바이너리
CMD ["./server"]
```

### 각 줄이 왜 이렇게 되는가

- **FROM `golang:1.22`** — 빌드의 시작점. 이 이미지에는 Go 컴파일러와 도구 전체가 들어 있어 용량이 큽니다(수백 MB~1GB). 지금은 "일단 도는 것"이 목표라 그대로 씁니다. 이 무게를 5번 문제에서 multi-stage build로 해결합니다.
- **WORKDIR `/app`** — `cd /app` 효과. 이후 `COPY`, `RUN`, `CMD`가 모두 이 경로 기준으로 동작합니다.
- **COPY go.mod → RUN go mod download → COPY . . → RUN go build** 순서가 핵심입니다. 모듈 정의를 먼저 복사해 의존성을 받아두면, 나중에 소스만 바뀌어도 의존성 다운로드 레이어가 **캐시로 재사용**됩니다. `COPY . .`를 위로 올리면 코드 한 줄만 고쳐도 매번 의존성을 새로 받습니다. (캐시 최적화는 3번에서 깊게 다룹니다.)
- **`go build -o server .`** — 현재 디렉터리의 소스를 컴파일해 `server`라는 실행 바이너리를 만듭니다. Go는 이렇게 **단일 바이너리**로 떨어지는 게 특징이고, 이 성질이 5번 경량화의 핵심 재료가 됩니다.
- **EXPOSE 5000** — 실제로 포트를 여는 게 아니라 "이 컨테이너는 5000을 쓴다"는 표시입니다. 외부 노출은 다음 단계의 `-p`가 담당합니다.
- **CMD `["./server"]`** — 컨테이너가 뜰 때 실행할 명령. 이 바이너리가 포그라운드로 계속 떠 있어야 컨테이너도 살아 있습니다.

## 2. 빌드

`app/` 디렉터리(또는 그 상위)에서 빌드합니다. 마지막 인자가 **빌드 컨텍스트**입니다.

```bash
docker build -t counter:1.0 ./app
```

`-t counter:1.0`으로 `[이미지이름]:[태그]`를 붙입니다. 태그를 생략하면 자동으로 `latest`가 붙습니다.

```bash
docker images counter
```

> 만들어진 이미지 용량을 눈여겨봐 두세요. 보통 수백 MB 이상으로 꽤 큽니다. 5번 문제에서 이걸 확 줄이게 됩니다.

## 3. 실행

```bash
docker run -d --name {{my-container}} -p 8080:5000 counter:1.0
```

- `-d` — 백그라운드(detached) 실행
- `--name {{my-container}}` — 컨테이너 이름 지정 (`{{ }}`는 여러분이 직접 정하는 값. 예: `web`). 이후 명령에서 ID 대신 이 이름을 씁니다
- `-p 8080:5000` — **호스트 8080 → 컨테이너 5000**. 왼쪽이 바깥(호스트), 오른쪽이 안(컨테이너)입니다. 서버가 5000에서 도니 오른쪽은 반드시 5000이어야 합니다.

## 4. `-it`와 `exec`에 대하여

검증 5)에서 `docker exec -it`로 컨테이너 안에 들어갔습니다. 이 옵션을 짚고 갑니다.

- **`-i`(interactive)** 는 표준입력(STDIN)을 열어두고, **`-t`(tty)** 는 터미널을 할당합니다. 셸처럼 사람이 타이핑하며 대화하는 작업엔 둘 다 필요해 보통 `-it`로 함께 씁니다. `-it` 없이 `docker exec {{my-container}} sh`를 치면 셸이 입력을 못 받고 즉시 빠져나옵니다.
- **`run`이 아니라 `exec`인 이유** — 컨테이너는 이미 `./server`를 띄운 채 돌고 있습니다. 그러니 **새로 만드는** `run`이 아니라 **돌고 있는 안으로 들어가는** `exec`를 씁니다. `exec`는 컨테이너 안에 새 프로세스(여기선 `sh`)를 하나 더 띄우는 것이라, `exit`로 나와도 원래 프로세스(PID 1, `./server`)는 그대로라 컨테이너는 계속 살아 있습니다.
- 참고로 `run`에 `-it`를 안 붙였어도 `exec -it`는 됩니다. `run`의 `-it`는 컨테이너 메인 프로세스(`./server`)에, `exec`의 `-it`는 지금 새로 띄우는 `sh`에 각각 적용되는 별개 옵션입니다. `exec`가 되려면 필요한 건 `run -it`가 아니라, 컨테이너가 지금 돌고 있다는 사실(`docker ps`에 `Up`)뿐입니다.
